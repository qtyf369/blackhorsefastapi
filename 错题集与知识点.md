# 3. yield 和 return 的区别，begin事务

```python
async def get_db():
    async with async_session_local() as session:
        with session.begin():  # begin事务，会自动commit，自动处理错误，自动回滚事务。
            yield session  # yield保持会话挂起，直到依赖项使用完毕
```

如果用return，会立即返回，不会保持会话挂起，导致数据库操作失败。
这里用begin,会自动commit事务。如果错误，会自动回滚事务，且抛出异常。由调用者来处理异常。
如果没用begin，需要手动commit事务，手动处理异常。如果错误，需要手动回滚事务。

# 4. 原子性

```python
 await db.execute(update(News).where(News.id == id).values(views=News.views + 1))
```

原子性（Atomicity） 是数据库事务的 ACID 属性之一，它保证一个事务中的所有操作要么全部成功提交，要么全部失败回滚，不会出现“执行了一半”的状态。
UPDATE news SET views = views + 1 WHERE id = ? 的 SQL，由数据库原子执行，避免并发问题。
如果先select，再update，会先查询到旧的views值，再更新，导致更新失败。如果用db.get(News, id)来查询再更新，同样会有并发问题。

# 8.索引

索引是一种优化数据库查询的机制。

from sqlalchemy import Index

# 9.接口参数

路径参数、查询参数和请求体，在 HTTP 请求中是完全不同的部分。如果不加说明，FastAPI 需要有一套明确的规则来区分它们。

FastAPI 的识别规则是：

如果参数在路径模板中（如 /items/{item_id}），它默认是路径参数。

如果参数是 `Pydantic BaseModel` 的子类，它默认是请求体参数。

如果参数是简单的 Python 类型（如 int, str），它默认是查询参数。

因此，当你希望一个简单类型的参数从请求体中获取时，就需要显式地“告诉” FastAPI，例如 `importance: int = Body(...)`，这能避免 FastAPI 将它错误地归类为查询参数。

Path 和 Query 的本质
首先是“标注”：它们告诉 FastAPI 这个参数应该从路径（Path）或查询字符串（Query）中提取，而不是从请求体或其它地方。这是一种来源标注。

其次是“限制”：它们可以附加验证规则（如 min_length, gt, regex 等），对提取到的数据进行校验。但这些限制是额外的，即使不加它们，只要参数名匹配，FastAPI 也能正确提取（路径参数需要写在装饰器 {} 中）。

Path 的使用范围
Path 只能用于路径参数。如果你试图在查询参数上使用 Path，FastAPI 会报错或忽略（不推荐）。路径参数必须同时满足：

装饰器路径中包含 {param_name}。

函数参数中使用 Path() 或直接声明（默认会识别为路径参数）。

为什么要用 Path 或 Query？
默认行为：如果一个参数满足路径变量（在 {} 中），即使不写 Path()，FastAPI 也会识别为路径参数。但如果你需要添加验证（如 min_length）或元数据（如 title），就必须显式使用 Path()。

查询参数同理：不写 Query() 时，简单类型且不是路径参数，会自动识别为查询参数。但当需要验证或文档描述时，用 Query()。

总结
Path 和 Query 的主要作用是标注来源，同时提供数据验证和描述。

Path 只能用于路径参数；Query 只能用于查询参数。

如果不需要验证，可以省略它们，FastAPI 会自动识别（但路径参数必须写在装饰器路径中）。

# 10.Field

`Field` 是一个工具函数，用于为模型字段添加额外的元数据、验证规则和默认值。它通常与 BaseModel 的字段类型注解一起使用。

```python
from pydantic import BaseModel, Field

class Item(BaseModel):

```

这些 description 会显示在 FastAPI 自动生成的 Swagger UI 中.

```python
age: int = Field(..., gt=0, le=150)           # 1~150
title: str = Field(..., min_length=1, max_length=50)
email: str = Field(..., regex=r'^[^@]+@[^@]+\.[^@]+$')
```

# 12. scalar_one_or_none() 和 scalar()，scalars()

`scalar_one_or_none()` 方法返回查询结果的第一个元素，如果查询结果为空则返回 None。如果查询结果有多个元素，会报错。
`scalar()` 方法返回查询结果的第一个元素，如果查询结果为空则返回 None。

```python
user = await db.execute(select(User).where(User.username == username))
user = user.scalar()
```

如果查询结果为空，`scalar()` 方法会返回 None。

### `result.scalars()` —— 取“多个”标量值

* **作用**：返回一个 `ScalarResult` 对象，它代表**所有行**的**第一列**的集合。

* **返回值**：`ScalarResult` 对象（本身是可迭代的），通常后续会调用 `.all()`、`.first()` 或直接迭代。

* **特点**：它也不返回 `Row`，而是返回一个专门用来遍历标量值的“迭代器”。

### 📊 关键区别对比表

| 方法              | 返回类型                   | 取第几行    | 取第几列    | 代表含义          |
| --------------- | ---------------------- | ------- | ------- | ------------- |
| **`scalar()`**  | `int` / `str` / ORM 实体 | **第一行** | **第一列** | **单数**（只要一个值） |
| **`scalars()`** | `ScalarResult` 对象      | **所有行** | **第一列** | **复数**（要一组值）  |

# 14. passlib 与哈希

`passlib` 是一个密码哈希库，用于安全地存储和验证密码。下面的用法已经包含了盐、成本因子和哈希值。

```python
from passlib.context import CryptContext
context = CryptContext(schemes=["bcrypt"], deprecated="auto") # 创建一个密码上下文对象，指定使用bcrypt算法。deprecated="auto" 表示自动选择最新的算法。bcrypt是一种安全的密码哈希算法。
encrypted_password = context.hash(userdata.password)
```

哈希化是一个同步操作，会阻塞代码，如果需要异步，可以使用 `asyncio.to_thread` 函数，放到线程池中执行。

```python
import asyncio
from passlib.context import CryptContext

context = CryptContext(schemes=["bcrypt"], deprecated="auto")

async def get_password_hash(password: str) -> str:
    # 在线程池中执行同步阻塞的哈希操作
    return await asyncio.to_thread(context.hash, password)
```

# 17.flush() 方法

在需要返回新创建对象 ID 给前端的场景，几乎必须使用 flush（或 commit，但后者会提交事务）。flush() 方法会发送 INSERT、UPDATE、DELETE 语句到数据库，但不会提交事务。

可以被 `rollback()` 方法取消。

在 SQLAlchemy 2.x 中，推荐的做法是在 async with session.begin(): 块内：

```python
db.add(obj)
await db.flush()    # 发送 INSERT，获取自增 ID
await db.refresh(obj) # 加载其他数据库生成的值
return obj          # 退出 begin 时自动 commit
```

如果不使用 `flush()`，`obj.id` 在 `commit` 之前一直是 `None`，前端无法获得新 ID。

# 18. default 和 server_default

`default` 是在创建对象时设置的默认值，`server_default` 是在数据库中设置的默认值。
`default` 是python端默认值，不会在数据库中设置默认值。
`server_default` 是数据库端默认值。有default从句，会在数据库中设置默认值。

## datetime.now(), func.now() 的配合使用

_*注意*：datetime.now() 是在python端调用的函数，返回当前时间。加括号了，会马上执行。如果在类中定义，不能加括号。否则所有实例的默认值都是当前时间。 func.now()返回的是一个对象，会在数据库中动态计算，所以一直是当前时间。_

一般用 default=datetime.now() 来设置默认值。func.now() 是在服务器端调用的函数，返回当前时间。
如果 `default`=func.now() 其实还是在数据库计算，但是数据库不会标注default，不如server_default。
datetime.now() 是在python端调用的函数，返回当前时间。
以下两个例子：

```python
    # Python 端默认值（由 SQLAlchemy 填充）
    created_at = Column(DateTime, default=datetime.now)
    # 数据库端默认值（由数据库填充）
    updated_at = Column(DateTime, server_default=func.now())
```

created_at：插入时如果未提供，SQLAlchemy 会调用 datetime.now() 填入 SQL 语句。缺点是**数据库不知道默认值**，直接执行 INSERT 而不经过 ORM 时，该列会变成 NULL（除非你手动提供）。

updated_at：表定义中有 DEFAULT CURRENT_TIMESTAMP，无论通过 ORM 还是直接 SQL，都会由数据库自动填充。

# 20. Optional

```python
nickname: Optional[str] = Field(None, max_length=50, description="用户昵称")
```

`Optional[str]` 表示这个字段可以是 str 类型，也可以是`None`。在 Pydantic 中，字段的默认值如果为 `None`，并且类型注解为 `Optional[str]`，那么该字段就不是必填的，客户端可以不传。如果类型注解是 `str` 而没有 `Optional`，即使默认值是 `None`，Pydantic 也会将其视为必填（因为类型注解不允许 `None`）。但更准确地说，Pydantic 判断字段是否必填的依据是有无默认值，以及类型是否允许 `None`。

**通常写法`Optional[str] = None` 表示该字段可选，默认值为 `None`。**

在SQLachemy中，等效于 `nullable =true`。

但在pydantic中，没有`nullable =true`,只能用Optional 表示 可不传。

# 21.响应体序列化

也就是封装返回体。

流程通常是：

1. 数据库查询 → 得到 SQLAlchemy ORM 实例（例如 User）

2. 将 ORM 实例转换为 Pydantic 模型实例（例如 UserOut）

3. FastAPI 自动将 Pydantic 实例序列化为 JSON 响应

## 当 Pydantic 模型与 ORM 模型字段不一致时

* 如果 Pydantic 模型有 ORM 模型没有的字段，你需要手动提供值（例如聚合计算、关联对象）。

* 如果 ORM 模型有字段但 Pydantic 模型没有（例如密码），那么那些字段不会被返回，达到了隐藏敏感信息的目的。

* 如果字段名不同，可以在 Pydantic 模型中使用 `Field(alias="db_column") `或配置 `from_attributes `时自定义属性名。

让 Pydantic 模型的约束与数据库/ORM 模型的约束完全一致或更宽松。

# 21. model_config

这是Pydantic 中， Basemodel 类的配置项。下面代码中，使用了

`model_config= ConfigDict(from_attributes=True)`

这个配置的作用是

**从其他对象（如 SQLAlchemy ORM 实例、字典、其他数据类）创建 Pydantic 模型对象。**

默认是False，那么只接受字典

在接口中，指定一个Basemodel的子类，return的结果会按照这个类的字段去解析。

`response_model=UserOut`

```
from pydantic import BaseModel, ConfigDict
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

# ORM 模型
class UserORM(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]

# Pydantic 模型
class UserOut(BaseModel):
    id: int
    name: str
    model_config = ConfigDict(from_attributes=True)

# 路由中
@router.get("/users/{user_id}", response_model=UserOut)
async def get_user(user_id: int, db: AsyncSession = Depends(get_db)):
    user = await db.get(UserORM, user_id)   # ORM 实例
    return user   # 自动转换
```

### 用途

将 SQLAlchemy ORM 对象（或其他任意具有属性的对象）直接转换为 Pydantic 模型，而无需手动写成字典。

在 FastAPI 中，当你为路由指定 response_model 为某个 Pydantic BaseModel 子类时，FastAPI 会自动完成以下工作：

1. 数据验证：检查你实际返回的数据是否符合 Pydantic 模型的字段类型、约束（如 max_length、ge 等）。如果不符，会自动返回 422 错误。

2. 数据过滤：只包含 Pydantic 模型中定义的字段，即使你返回了额外的字段（例如 ORM 模型中的密码字段），也会被自动排除。

3. 类型转换：将返回的数据（如 datetime、UUID、自定义类）转换为 JSON 兼容的类型，如果有必要还会调用 model_dump()。

4. 自动文档生成：根据 Pydantic 模型的字段定义生成 OpenAPI 文档。

### 但需注意：

* 如果你返回的不是 Pydantic 模型实例，而是字典或 ORM 对象，FastAPI 会隐式地调用 model_validate 或使用 from_attributes（如果配置了）进行转换。因此，返回的原始对象不一定是 Pydantic 模型实例，但最终响应会按照 response_model 的形状输出。

* 为了避免混淆，通常建议直接返回 Pydantic 模型实例，或者至少确保返回的数据能够被转换为 response_model 指定的类型。

# 22. JSONResponse

JSONResponse 是 FastAPI（以及底层的 Starlette）中用于专门返回 JSON 格式响应的一个类。它的作用是：

1. 确保响应内容类型为 application/json：它会自动设置 Content-Type 头为 application/json。

2. 自动序列化 Python 对象为 JSON 字符串：你传入的 Python 对象（如字典、列表、Pydantic 模型等）会被转换成 JSON 字符串。

3. 提供更灵活的控制：相比直接返回字典，JSONResponse 可以让你自定义状态码、响应头等。

## 与直接返回字典的区别

* 直接返回字典：`return {"message": "hi"} `→ FastAPI 自动包装成 JSONResponse，状态码默认为 200。

* 使用 JSONResponse：允许你显式控制状态码、响应头等，且内容可以是任意可序列化的对象。

## 注意

如果已经在接口返回了`JSONResponse`，`response_model`将被忽略

`JSONResponse`的 `content` 参数通常传入 字典 或 列表，但也可以是任何可 JSON 序列化的 Python 类型（如字符串、数字等）。然而，Python 标准库的 `json.dumps` 不能直接处理 `datetime`、`UUID`、`Decimal`或自定义对象。因此，当你手动创建 `JSONResponse` 时，如果 `content` 中包含这些非 JSON 兼容的类型，就会抛出 `TypeError`。这时就需要使用 `fastapi.encoders.jsonable_encoder` 先将其转换为兼容类型，再传入 `JSONResponse`。

# 23. .model_validate()

`.model_validate()` 是 Pydantic V2 中用于从各种输入（字典、对象、JSON 字符串等）创建并验证 Pydantic 模型实例的核心方法。

## 主要用途

* 从字典创建模型实例

* 从其他对象（如 SQLAlchemy ORM 实例）创建模型实例（需配合 `from_attributes=True`）

* 从 JSON 字符串创建模型实例（使用 `.model_validate_json()`）

## 基本用法

```python
from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str

# 从字典创建
user = User.model_validate({"id": 1, "name": "张三"})

# 从关键字参数创建（通常直接用构造函数）
user = User(id=1, name="张三")   # 等价，但更简洁
```

## 与 from_attributes 配合

```python
from pydantic import BaseModel, ConfigDict

class UserOut(BaseModel):
    id: int
    name: str
    model_config = ConfigDict(from_attributes=True)

# 假设有 ORM 实例
user_orm = UserORM(id=1, name="张三")

# 从 ORM 实例创建 Pydantic 模型
user = UserOut.model_validate(user_orm)   # 自动读取 user_orm.id 和 user_orm.name
```

## 其他相关方法

* .model_validate_json(json_str)：从 JSON 字符串创建。

* .model_dump()：将模型转为字典。

* .model_dump_json()：将模型转为 JSON 字符串。

## 总结

* `.model_validate()` 是 Pydantic V2 中推荐的从各种数据源创建并验证模型的统一入口。

* 它与 `from_attributes` 配置一起使用，能够方便地从 ORM 对象等非字典源构建模型。

* 在 FastAPI 中，通常你不需要直接调用 `.model_validate()`，因为 FastAPI 会自动根据` response_model` 和请求体做验证。但在手动转换数据时非常有用。

# 21. 抽离错误处理

## 1️⃣ 确定需要处理的异常类型

根据项目需求，列出预期可能发生的异常：

* 数据库异常：sqlalchemy.exc.IntegrityError（唯一约束、外键冲突等）

* FastAPI 内置异常：HTTPException（通常用于手动返回错误）

* 自定义业务异常（例如 UserNotFoundError、PermissionDenied）

* 其他意外异常：Exception（作为兜底）

## 2️⃣ 编写异常处理函数

每个处理器是一个普通函数（或异步函数），接收 (request: Request, exc: Exception) 参数，返回 JSONResponse 或 Response 对象。

### 

## 3️⃣ 将处理器注册到 FastAPI 应用

使用 @app.exception_handler 装饰器，将上述函数绑定到具体的异常类。注意注册顺序：应该先注册更具体的异常，后注册更通用的异常（例如先 IntegrityError，再 Exception）。FastAPI 会按照注册顺序从上到下匹配，一旦匹配就不会继续往下。

## 4️⃣ 集中管理（推荐）

如果项目较大，可以把所有异常处理器放在一个独立的模块（例如 handlers/exception_handlers.py）中，然后定义一个函数来注册它们：

```python
#exception_handlers.py
from fastapi import FastAPI
from .handlers import integrity_handler, http_exception_handler, generic_handler

def register_exception_handlers(app: FastAPI):
app.add_exception_handler(IntegrityError, integrity_handler)
app.add_exception_handler(HTTPException, http_exception_handler)
app.add_exception_handler(Exception, generic_handler)

```

然后在main.py 中调用：

```Python
from exception_handlers import register_exception_handlers

app = FastAPI()
register_exception_handlers(app)
```

## 5️⃣ 在路由中使用

注册后，路由中可以直接 raise 相应的异常，处理器会自动捕获：

```Python
@router.post("/register")
async def register(...):
    if user_exists:
        raise HTTPException(status_code=400, detail="用户名已存在")
    # 如果数据库 IntegrityError 发生，会被 integrity_handler 捕获
```

无需在路由中写 `try...except`（除非需要额外处理）。

# 23. Fastapi的错误捕获

**在 FastAPI（以及任何基于 ASGI/异步的 Web 框架）中，单个请求处理过程中抛出的未捕获异常，不会导致整个服务器进程崩溃。事件循环会捕获该协程的异常，记录错误日志，然后继续处理其他请求。因此，捕获异常的主要目的已经不再是“防止服务器崩溃”，而是：**

1. 返回友好的错误响应（提供具体的错误原因，而不是模糊的 `500 Internal Server Error`）。

2. 记录详细的错误日志（便于开发者排查问题）。

3. 确保资源正确释放（例如数据库会话回滚、文件关闭等，但这些通常由上下文管理器保证）。

# 24. Header

## Authorization: str = Header(None)

* 手动从请求头中提取 Authorization 字段的值，得到的是完整的字符串（例如 "Bearer eyJhbGciOiJIUzI1NiIs..."）。

* 你需要自己验证格式是否为 Bearer ，并提取 token 部分。如果头不存在或格式错误，你需要手动处理错误响应（例如抛出 HTTPException）。

* 不会自动生成 OpenAPI 文档中的安全方案，需要额外配置 security 参数。

## HTTPBearer() + Depends

* 自动解析 Authorization 头，验证格式，并提取 token 字符串。

* 自动处理缺失或格式错误的请求头，返回 403 错误。

* 自动生成 OpenAPI 文档，并显示“Bearer Token”认证方案。

* 代码更简洁，符合标准做法。

## 结论

除非你有非常特殊的需求（例如自定义认证方案），否则应该使用 HTTPBearer()。它更符合 FastAPI 的设计哲学，也更安全、更简洁。直接写 = Header() 虽然可以工作，但会失去 FastAPI 提供的许多便利，不推荐在生产项目中使用。

# 25. 泛型响应模型

from pydantic import BaseModel
from typing import Generic, TypeVar, Any

T = TypeVar("T") #创建一个类型变量T，左边是变量名，右边是字符串参数，用于调试。当你在错误信息或类型提示中看到这个类型变量时，会显示这个名字（例如 "T"）。

# 27. 数据库时间

时间存入数据库不保存时区。

我尝试用一个**没有时区信息的 `datetime` 对象**（offset-naive）与一个**有时区信息的 `datetime` 对象**（offset-aware）进行了比较（例如 `>`、`<`、`==`）。Python 禁止这种比较，因为无法确定两个时间是否在同一时区基准下。

### 统一为 offset-aware（推荐）

```python
from datetime import datetime, timezone

# 假设 db_time 是 naive，且实际表示 UTC（但未标记）
# 可以为其添加 UTC 时区信息
db_time_aware = db_time.replace(tzinfo=timezone.utc)

# 然后与 aware 时间比较
now_aware = datetime.now(timezone.utc)
if db_time_aware < now_aware:
    ...
```

## replace(tzinfo=timezone.utc) 做了什么？

* 它不改变时间数值，只是**给这个 naive datetime 加上了“UTC”时区标签**，告诉 Python：这个时间的解释是 UTC 时间。

* 然后你就可以将其与另一个 aware datetime（如 `datetime.now(timezone.utc)`）进行比较，因为两者都有时区标签，且都是 UTC。

# 28. PUT 和 PATCH

PUT：请求体必须包含资源的完整表示。例如更新用户信息时，客户端需要发送所有字段（包括不需要修改的字段）。

PATCH：请求体只包含需要修改的字段。

PUT：客户端需要提前获取完整的当前资源，然后修改部分字段，再全量发送。需要额外的 GET 请求。

PATCH：客户端只需发送变更的字段，无须知道资源的完整状态。在网络条件较差或频繁修改少量字段的场景下更高效。

教程为了简化代码（避免区分两种方法的实现逻辑），采用了 PUT + exclude_unset 的模式，本质上实现了 PATCH 的行为。这在功能上没有问题，但不符合 RESTful 规范。如果你希望严格遵循规范，应改用 PATCH 方法，并确保前端按部分更新的方式发送请求。

# 29. 自定义异常

如果自定义异常类只写了 `pass`，那么实例化时传入的消息可以通过 `str(exc)` 或 `exc.args[0]` 获取，因为父类 Exception 已经实现了这些机制。

### 原理

Exception 的 **init** 方法会接收任意数量的参数，并将它们存储在 self.args 元组中。

str(exc) 会返回 self.args 中第一个元素的字符串表示（如果只有一个参数的话）。

即使子类体是空的（pass），它仍然继承了父类的这些行为。

```python
class UserNotFoundError(Exception):
    pass

# 抛出异常
raise UserNotFoundError("用户 admin 不存在")

# 捕获异常
try:

except UserNotFoundError as e:
    print(str(e))          # "用户 admin 不存在"
    print(e.args[0])       # "用户 admin 不存在"
    print(e.args)          # ("用户 admin 不存在",)

```

## 在全局异常处理器中的应用

```python
@app.exception_handler(UserNotFoundError)
async def user_not_found_handler(request: Request, exc: UserNotFoundError):
    # 方式一：使用 str(exc)
    message = str(exc)  # "用户 admin 不存在"

    # 方式二：直接取 args 第一个元素
    message = exc.args[0] if exc.args else "用户不存在"

    return JSONResponse(
        status_code=404,
        content={"code": 404, "message": message}
    )
```

# 33. update 和 delete

| 操作             | 针对单个对象（ORM 实例）                    | 针对批量（Core 语句）                      |
| -------------- | --------------------------------- | ---------------------------------- |
| **更新（Update）** | `user.name = "新"` + `db.commit()` | `update(表).where(...).values(...)` |
| **删除（Delete）** | `await db.delete(user)`           | `delete(表).where(...)`             |
| **执行方式**       | 通过 `db`（会话）                       | 通过 `db.execute(stmt)`              |
| **是否返回数据**     | 修改实例本身                            | 返回影响行数（`result.rowcount`）          |

| 操作     | 针对单个对象（实例法）                             | 针对批量（Core 语句）                                                |
| ------ | --------------------------------------- | ------------------------------------------------------------ |
| **删除** | `await db.delete(user)`                 | `await db.execute(delete(User).where(...))`                  |
| **更新** | `user.name = "新"` + `await db.commit()` | `await db.execute(update(User).where(...).values(name="新"))` |

* **`db.delete(instance)`**：直接删除实例，无需指定值。

* **修改实例属性**：没有专门的 `db.update`，而是通过修改属性 + `commit` 触发更新。

* **批量更新**：使用 `update()` 构造器，必须通过 `.values()` 设置新值。
  
  

`result=await db.execute(delete(User).where(...))` 

`result.rowcount`是实际被删除的行数,update的result也是这么用的。

# 35. 外键 与 relationship

### 🎯 黄金法则（90% 的情况）

1. **如果外键在这个表里**（比如 `FavourtieNews` 里有 `user_id`）：
   
   * 这个表是 **“多”** 的一方。
   
   * 它指向的对方（`User`）是 **“一”** 的一方。
   
   * 所以，这个表里的关联字段是 **单个对象**（`Mapped["User"]`）。

2. **如果外键在别的表里**（比如 `FavourtieNews` 里有 `user_id`，但你现在是在 `User` 表里）：
   
   * 这个表（`User`）是 **“一”** 的一方。
   
   * 它被对方（`FavourtieNews`）指向。
   
   * 所以，这个表里的关联字段是 **列表**（`Mapped[list["FavourtieNews"]]`）。

* * *

### 🤔 如果不是“一”和“多”，而是“多对多”呢？

还有一种常见情况是 **“多对多”**（Many-to-Many），比如“学生”和“课程”。这时会有一个中间表，两边都没有直接的外键。

* 这种情况下，**两边都是“多”方**，两边都是列表：
  
  * `Student.courses: Mapped[list["Course"]]`
  
  * `Course.students: Mapped[list["Student"]]`

### 💡 总结一句话

> **“持有外键的那一侧，永远是单数；不持有外键的那一侧，永远是复数（多对多除外，两边都是复数）。”**



```python
news: Mapped["News"] = relationship("News", back_populates="news_favourites") 
```

` back_populate` 需要两边都写或都不写，两边写是双向绑定。

### 🧩 三种写法对比

| 写法                                                               | 结果                                         | 适用场景                               |
| ---------------------------------------------------------------- | ------------------------------------------ | ---------------------------------- |
| **A 端写 `back_populates`，B 端没写**                                  | ❌ **启动报错**（ArgumentError）                  | 错误用法                               |
| **A 端写 `relationship`，不带 `back_populates`，B 端不写**                | ✅ 单向关系，`news.favourites` 可用，`fav.news` 不可用 | 只需要单向访问                            |
| **A 端写 `back_populates="xxx"`，B 端写 `back_populates="yyy"`，互相配对** | ✅ 双向关系，两端都能访问                              | 需要双向导航 |
